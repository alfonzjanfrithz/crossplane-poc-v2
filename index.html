Crossplane v2 PoC fully v2-native

A from-scratch, fully v2-native Crossplane proof of concept on a local kind + podman cluster against MiniStack + Vault. A producer app creates a namespaced composite resource (XBucket); Crossplane composes an S3 bucket (via the v2 native per-service AWS provider), materialises the dynamic bucket name into a Secret, and — inside the same composition — uses an ESO PushSecret to publish that name to Vault. Putting the Vault write in the composite makes it lifecycle-coupled: when the XR is deleted, the composed PushSecret is deleted too and ESO removes the Vault entry, so no secret outlives its resource. A second app in a separate namespace then reads that value back out of Vault via an ESO ExternalSecret and accesses the same shared bucket — proving cross-namespace resource + secret sharing with Vault as the substrate.

One command brings it all up: ./scripts/up.sh

Architecture

  • Three namespaces: demo-app (producer), demo-app-2 (consumer), rss (Vault).
  • Producer creates a namespaced XBucket XR; Crossplane composes a real S3 bucket in MiniStack (via the v2 native per-service AWS provider), a native Secret carrying the dynamic bucket name, and an ESO PushSecret.
  • Dynamic value → Vault, in-composite: the composed PushSecret writes the bucket name to Vault at a stable, consumer-knowable path (crossplane/<xr-name>-bucket). It is lifecycle-coupled — the entry is deleted with the bucket.
  • Consumer pulls it back from Vault via an ESO ExternalSecret in a separate namespace, then reads the same shared bucket.
  • Vault is the sharing substrate — no direct cross-namespace Secret access; the value round-trips through Vault.
  • MiniStack runs outside the cluster on podman, reached via an in-cluster headless Service + manual Endpoints.
  • Fully v2-native: namespaced XRs and MRs (s3.aws.m.upbound.io), no claims, composed resources inherit the XR's namespace.

Seams: the linkage snippets

The whole system is just four wiring points. These are the minimal snippets at each seam, in data-flow order.

Seam 1 — one expression computes the dynamic name and threads it everywhere (crossplane/composition.yaml). The same value (<prefix><XR-name><XR-uid>) is stamped into the Bucket's external name and the Secret's bucketName, which the PushSecret (Seam 2) then pushes to Vault.

- type: CombineFromComposite
  combine:
    strategy: string
    variables:
      - fromFieldPath: spec.bucketPrefix
      - fromFieldPath: metadata.name      # XR name
      - fromFieldPath: metadata.uid       # random + stable
    string:
      fmt: "%s%s-%s"
  toFieldPath: metadata.annotations[crossplane.io/external-name]

Seam 2 — composed PushSecret → Vault, ESO (lifecycle-coupled) (crossplane/composition.yaml, the vault-secret PushSecret). An ESO PushSecret reads the composed Secret and pushes its bucketName into Vault. The Vault key is built from the XR's stable metadata.name (crossplane/<xr-name>-bucket); the value is the dynamic bucket name from Seam 1. It is a composed resource, so it shares the XR's lifecycle, and deletionPolicy: Delete makes ESO remove the entry when the XR is deleted.

- type: FromCompositeFieldPath
  fromFieldPath: metadata.name              # XR name -> stable Vault key
  toFieldPath: spec.data[0].match.remoteRef.remoteKey
  transforms:
    - type: string
      string: { type: Format, fmt: "crossplane/%s-bucket" }
spec:
  deletionPolicy: Delete                       # value removed from Vault on XR delete
  secretStoreRefs:
    - { name: vault-backend, kind: SecretStore }
  selector:
    secret: { name: <xr-name>-bucket }         # the composed Secret (Seam 1)
  data:
    - match:
        secretKey: bucketName
        remoteRef: { remoteKey: crossplane/<xr-name>-bucket, property: bucketName }

Seam 3 — Vault → consumer Secret, ESO pull (charts/demo-app-2/templates/eso.yaml). The inverse of the PushSecret write (Seam 2), in a different namespace: an ExternalSecret reads the stable key back out of Vault into a local Secret — closing the loop cross-namespace (demo-app-2 never touches demo-app).

target:
  name: shared-bucket
  creationPolicy: Owner
data:
  - secretKey: bucketName
    remoteRef:
      key: crossplane/demo-app-bucket         # the stable key Seam 2 wrote
      property: bucketName

Seam 4 — consumer consumes the local Secret → shared bucket (charts/demo-app-2/templates/deployment.yaml). The pod cannot start until shared-bucket exists (Seam 3), so its readiness self-proves the chain.

env:
  - name: BUCKET_NAME
    valueFrom:
      secretKeyRef:
        name: shared-bucket
        key: bucketName

Throughline: Seam 1 threads the dynamic name into the Bucket and the Secret. Seam 2 is the single in-composite Vault write — an ESO PushSecret, lifecycle-coupled, so the secret dies with the bucket — using the XR's stable name as the key. Seam 3 is the pull that closes the loop in another namespace; Seam 4 is the app using it.

The hard-won v2 gotchas

1. Host inotify limit — provider-aws-s3 watches ~48 S3 CRDs (each an inotify instance); the kernel default fs.inotify.max_user_instances=128 is exhausted and the controller crashloops with too many open files. Rootless podman/kind cannot change it from inside the node — it must be raised on the host: 
sudo sysctl -w fs.inotify.max_user_instances=8192 fs.inotify.max_user_watches=1048576
2. Family-config RBAC gap — the per-service provider must read the provider-family-aws CRDs (clusterproviderconfigs, providerconfigs, providerconfigusages) but its auto-RBAC omits them. A manual ClusterRole grant is required, bound to the provider's SA.
3. Stable SA via DeploymentRuntimeConfig — revision SAs are hashed and change, breaking the RBAC binding. A DeploymentRuntimeConfig pins a stable provider-aws-s3 ServiceAccount.
4. endpoint.source: Custom + endpoint.services: [s3] — both required. Without source: Custom the SDK uses AWS service metadata; without services the override applies to no service. In either case traffic silently goes to real AWS (403 InvalidAccessKeyId) and MiniStack logs show nothing.
5. Credentials source: Secret — source: None yields empty creds (no SDK-chain fallback); there is no Environment source. Use source: Secret with an INI body.
6. spec.crossplane.compositionRef — v2 moved the XR's Crossplane machinery under spec.crossplane.
8. Composing a non-provider CRD (PushSecret) needs aggregated RBAC — Crossplane's composite controller composes resources using the core crossplane ServiceAccount. It has RBAC for core types (the composed Secret) and for installed providers' MRs, but not for arbitrary CRDs like ESO's PushSecret — composing one fails with cannot create PushSecret: ...forbidden. Fix: crossplane/eso-pushsecret-rbac.yaml, a ClusterRole labelled rbac.crossplane.io/aggregate-to-crossplane: "true" granting CRUD on pushsecrets.external-secrets.io (the PushSecret equivalent of the family-config RBAC in gotcha #2).
9. MiniStack rejects the AWS SDK v2 default checksum (CRC64NVME) — AWS CLI/SDK v2 (2024+) defaults to the CRC64NVME checksum on PutObject; MiniStack bundles only SHA256/SHA1/CRC32 (no native CRC64NVME/CRC32C, to keep the image ~270 MB). So aws s3 cp from a pod fails with InvalidRequest: Checksum algorithm not supported ... CRC64NVME while the provider’s CreateBucket (no object checksum) succeeds → the bucket is Ready but the app logs write FAILED. Fix: AWS_REQUEST_CHECKSUM_CALCULATION= when_required (+ AWS_RESPONSE_CHECKSUM_VALIDATION=when_required) on any pod calling S3 — already wired into both demo Deployments. (LocalStack bundles CRC64NVME, so this is a new MiniStack-specific trap.)

Lessons learned: what cost rework

Honest retrospective — each row caused an apply-fail-retry loop or a debugging detour.

#MistakeWhat happenedHindsight (faster path)
1Debugged auth before checking where the request landed~5 cycles on credentials (None → Environment, not in the enum → Secret) chasing InvalidAccessKeyIdTraffic was going to real AWS, not MiniStack. Ask "is the request reaching MiniStack?" first — one podman logs ministack during a reconcile. Tell: real-AWS-style RequestID/HostID + zero MiniStack logs.
2Re-extracted a schema, then ignored itHad the source enum + endpoint.services, still wrote source: Environment and omitted servicesAct on the schema you just dumped; don't re-guess from memory.
3Didn't fix the known resource limit before installingGambled "lean" install would dodge the inotify exhaustion; it didn't (aws-s3 alone needs ~48). Also tried sysctl inside the node (denied under rootless podman)Raise host fs.inotify.max_user_instances as a precondition, before installing any upjet provider.
4Iterated RBAC one permission at a timeread configs → then create on usages → then re-bind after SA changedThe upjet family gap is predictable: read-on-configs + CRUD-on-usages, bound to a stable SA. Write the full ClusterRole first try.
5Manually installed the family providerInstalled it on top of the revision aws-s3 auto-resolves, creating duplicates and polluting the clusterPer-service providers auto-resolve their family dependency; don't install it manually.
6Repeated v2 mistakes that were in my own notesspec.compositionRef (should be spec.crossplane.*) and HCL one-arg-per-lineApply the notes already written; slow down on the parts you "know".
7Reactive, one-error-at-a-time debuggingread → fix one thing → re-apply → read nextReason about the full dependency chain up front; ~3 iterations collapse into one.

Throughline on #1, #2, #4, #7: debugged reactively instead of verifying the lowest layer first (is the request even arriving?) and reasoning about the whole chain before changing anything.
